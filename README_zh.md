# 高性能 N-Body 引力模拟引擎

 [🇺🇸 English](README.md)

## 项目背景 (Background)

​	本项目旨在从零构建一个跨语言、高性能的 N-Body 万有引力模拟架构。在虚拟宇宙中生成几万至几十万个带有质量、速度和坐标的“星体”，由于星体之间两两都会产生万有引力，其纯数学计算量高达 $O(N^2)$。

​	如果直接使用 Python 算这个会直接卡死。因此，本项目摒弃了多余的网络通信层，采用**“底层极致算力 + 顶层双端渲染”**的硬核架构：

1. **核心计算引擎：** 纯 C++ 打造，榨干本地多核 CPU 与内存的极限性能，负责唯一的物理模拟真实来源（Source of Truth）。
2. **轻量级可视化（Python 端）：** C++ 引擎编译为动态链接库（.dll/.so），Python 通过直接读取物理内存加载调用该引擎。结合 OpenGL 库（如 VisPy/PyVista）实现基础的业务逻辑调度与 3D 鼠标拖拽交互。
3. **次世代可视化（UE5 端）：** 直接将 C++ 核心计算引擎封装为 Unreal Engine 5 的底层 Plugin。利用 UE5 的渲染管线，实现从纯粹的物理数学计算到商业级粒子光影渲染的跨越。

------

### 挑战任务拆解

​	你要用 C++ 从零手搓这个底层的计算核心，你需要解决以下六个核心问题：

**挑战 1：内存与数据结构（对应阶段一、三）**

- **需求：** 几十万个星体的数据，你不能用普通的数组存，因为数量是动态的。
- **考验：** 你需要设计一个星体的数据结构（`struct` 或 `class`），并使用 C++ 的标准模板库（如 `std::vector`）来在堆内存中动态管理这几十万个对象。你要保证在这个过程中，坚决不发生内存泄漏（使用 `std::unique_ptr` 等智能指针管理生命周期）。

**挑战 2：面向对象的设计（对应阶段二）**

- **需求：** 你需要一个总控的 `Universe` 类来管理所有星体，并且对外暴露简单的接口给 Python 或 UE5 调用。
- **考验：** 如何封装内部的复杂逻辑？如何写出高效的构造函数来初始化你那 16GB 内存里的一大块连续空间？

**挑战 3：现代 C++ 特性（对应阶段四）**

- **需求：** 在写复杂遍历和底层算法时，让代码既跑得快又好维护。
- **考验：** 运用 `auto` 简化冗长的类型声明，使用 Lambda 表达式把物理公式（$F = G \frac{m_1 m_2}{r^2}$）直接内联到计算循环中，极大减少函数调用的开销。

**挑战 4：榨干 i7 的多线程并发（对应阶段五 - 核心难点）**

- **需求：** 单线程算 $O(N^2)$ 太慢了。你的 i7-13620H 有 10 个核心（6个大核+4个小核）。你需要把计算任务切分。
- **考验：** 1. 你需要开辟一个自定义线程池（`std::thread`）。
  2. 让线程 1 算前一万个星体，线程 2 算第二万个……
  3. **致命危险：** 如果两个线程同时试图去修改同一个星体的位置数据，程序会直接崩溃（数据竞争）。你必须引入 `std::mutex`（互斥锁）或无锁原子操作来保证数据安全。

**挑战 5：Python 跨语言内存级调用（业务拓展）**

- **需求：** C++ 写好了，怎么让 Python 调它？
- **考验：** 你需要学习如何把 C++ 代码单独编译成动态链接库（`.dll` 或 `.so`）。然后通过 Python 的 `ctypes` 或者现代的 `pybind11` 库，打通这两个语言的任督二脉，实现零延迟的数据共享与业务逻辑控制。

**挑战 6：Unreal Engine 5 插件化封装（渲染终极拓展）**

- **需求：** 将脱离于任何引擎的纯 C++ 代码，无缝衔接进庞大的游戏引擎生态。
- **考验：** 学习和理解 UE5 的 `Build.cs` 模块化构建系统。将 C++ 引擎算出的海量坐标数据，安全地投喂给 UE5 的 Niagara 粒子系统或大量 Actor，在保持 60FPS 帧率的前提下完成影视级的渲染呈现。
